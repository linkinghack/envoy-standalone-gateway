# 01 - 初始系统功能分析

- **项目**: envoy-standalone-gateway
- **定位**: A general standalone API gateway & reverse proxy based on Envoy with pro admin UI
- **文档状态**: 初稿（Draft）
- **日期**: 2026-07-15

---

## 1. 背景与问题陈述

### 1.1 现状

Envoy 是当前能力最强的开源代理数据面之一：热更新不断连接、原生 HTTP/2 / HTTP/3 / gRPC、丰富的 L4/L7 过滤器体系、一流的可观测性。然而在实际使用中，Envoy 几乎被"锁死"在 Kubernetes / Service Mesh 生态里：

1. **配置门槛极高**。Envoy 原生 static 配置是面向机器生成而非面向人手写的，概念层级深（listener → filter chain → HCM → route config → virtual host → route → cluster → endpoint），普通运维/开发人员难以直接驾驭。
2. **现有控制面全部假设 k8s 环境**。Istio、Contour、Envoy Gateway 等控制面均以 Kubernetes CRD 为配置载体，脱离 k8s 无法工作。一台普通 Linux 服务器上的用户没有可用的 Envoy 控制面。
3. **配置抽象不友好**。Nginx 配置语法古老且概念晦涩；Istio 的 Gateway/VirtualService/DestinationRule 拆分模型对新手同样难以理解。行业缺少一套**平台中立、易理解、易扩展**的网关配置描述协议。
4. **专业管理控制台是商业软件专属**。查看实时生效的 clusters/listeners、流量统计、可视化配置管理，这类能力通常只在 NGINX Plus、Kong Enterprise 等收费产品中提供。

### 1.2 机会

把 Envoy 做成一个 **Nginx replacement** 级别的通用反向代理与 API 网关软件：

- 让不熟悉 Envoy 的用户通过简单抽象协议或 Web 控制台即可完成整套代理配置；
- 让熟悉 Envoy 的专家仍可直接使用原生 static 配置获得全部能力；
- 让 Envoy 在裸机、虚拟机、容器、k8s 中同样好用，不绑定任何平台；
- 把商业级管理控制台直接开源，做行业破局者。

## 2. 项目目标

### 2.1 目标（Goals）

| # | 目标 |
|---|---|
| G1 | 提供一个**轻量管理软件**（管理面），通过它控制 Envoy（数据面）实现用户的全部代理/网关配置 |
| G2 | 支持两种驱动 Envoy 的方式：**生成 static YAML 配置** 与 **xDS API 动态下发**，用户可按场景选择 |
| G3 | 设计一套**平台中立、易理解、易扩展的网关配置描述协议**（下称"抽象协议"），作为面向用户的首选配置模型 |
| G4 | 同时支持用户直接编写**原生 Envoy static 配置**，两种配置来源最终都编译为 Envoy 配置对象 |
| G5 | 提供**专业 Web 管理控制台**：配置管理、统计信息、Envoy 实时生效状态（clusters、listeners、routes、endpoints 等）可视化 |
| G6 | **轻量、易部署、易维护**：支持 Linux 原生运行（systemd）、Docker 直接运行、Kubernetes 运行，且不强依赖 k8s |
| G7 | 环境感知增强：检测到运行于 k8s 时，提供 Service 自动发现等便利能力（如配置 upstream 时列出集群内 SVC 供选择） |

### 2.2 非目标（Non-Goals，首期）

| # | 非目标 | 说明 |
|---|---|---|
| N1 | Service Mesh / sidecar 注入 | 不做 mesh，专注南北向网关与通用反向代理 |
| N2 | 替代 Istio / Envoy Gateway 的 k8s Ingress 实现 | k8s 只是部署环境之一，不以 CRD 为配置载体 |
| N3 | 多集群联邦、跨地域全局流量调度 | 远期再议 |
| N4 | 自研数据面 | 数据面完全复用 Envoy 官方发行版，不 fork、不修改 Envoy |
| N5 | 计费、多租户 SaaS 化能力 | 面向自托管场景 |

## 3. 用户画像与使用场景

### 3.1 用户画像

| 画像 | 描述 | 关键诉求 |
|---|---|---|
| P1 传统运维 / SRE | 长期使用 Nginx/HAProxy，管理若干台 Linux 服务器 | 像装 Nginx 一样简单地装好用起来；不想学 Envoy 概念；要 Web 控制台 |
| P2 后端开发者 | 需要给自己的服务快速配一个反向代理 / API 网关 | 简单的声明式配置，几分钟搞定路由、TLS、限流 |
| P3 Envoy 专家 / 平台工程师 | 深度了解 Envoy，需要精细控制 | 直接写原生 static 配置；用 xDS 做动态下发；能看到 Envoy 真实生效状态 |
| P4 中小团队技术负责人 | 需要商业网关的管理能力但没有预算 | 开源、免费、专业控制台、可长期自托管 |

### 3.2 典型场景

- **S1 单机反向代理（Nginx 替代）**：一台 VPS，systemd 拉起管理面 + Envoy，通过 Web UI 配置若干域名的 TLS 终止与后端转发。
- **S2 API 网关**：为一组后端服务配置统一入口，包含路由、路径重写、限流、鉴权（如 JWT）、CORS、超时/重试策略。
- **S3 Docker 单容器部署**：`docker run` 一条命令启动完整网关（管理面 + Envoy 同容器或 sidecar 容器）。
- **S4 k8s 内的独立网关**：以 Deployment 方式运行，管理面自动列出集群内 Service 作为 upstream 候选，但配置模型与 k8s 外完全一致。
- **S5 专家模式**：Envoy 老手直接粘贴/编辑原生 static 配置片段，管理面负责校验、版本管理与下发（static 文件或 xDS）。
- **S6 运行状态排障**：通过控制台实时查看当前生效的 listeners/clusters/endpoints 健康状态、请求统计，定位配置未生效或后端异常问题。

## 4. 系统总体形态

```
                          ┌────────────────────────────────────────┐
  用户                    │        管理面 (Management Plane)         │
  ├── Web 控制台  ──────►  │  ┌──────────────────────────────────┐  │
  ├── REST API   ──────►  │  │ 配置模型层                         │  │
  └── CLI (远期) ──────►  │  │  · 抽象协议 (首选)                  │  │
                          │  │  · 原生 Envoy static (专家)         │  │
                          │  ├──────────────────────────────────┤  │
                          │  │ 编译/校验层 → Envoy 配置对象         │  │
                          │  ├──────────────────────────────────┤  │
                          │  │ 下发层                             │  │
                          │  │  · static YAML 渲染 + 热重启协调     │  │
                          │  │  · xDS Server (ADS)                │  │
                          │  ├──────────────────────────────────┤  │
                          │  │ 状态采集层                          │  │
                          │  │  · Envoy admin API / stats 抓取     │  │
                          │  │  · k8s 环境感知 (可选增强)           │  │
                          │  └──────────────────────────────────┘  │
                          └───────────────┬────────────────────────┘
                                          │ static yaml / xDS / admin API
                                          ▼
                          ┌────────────────────────────────────────┐
                          │         数据面: 原生 Envoy               │
                          └────────────────────────────────────────┘
```

核心设计原则：

1. **单向编译**：所有面向用户的配置（抽象协议 / 原生 static）最终统一编译为内部 Envoy 配置对象模型（IR），再由下发层选择输出形态。抽象协议与下发方式解耦。
2. **数据面零侵入**：使用官方 Envoy 发行版，不做任何修改。
3. **k8s 是增强不是依赖**：所有 k8s 相关能力封装在可选的环境感知模块中，缺失时系统功能完整。

## 5. 功能需求

> 优先级标记：**P0** = 首个可用版本（MVP）必须；**P1** = 1.0 前完成；**P2** = 后续迭代。

### 5.1 FR-1 网关配置抽象协议

| ID | 需求 | 优先级 |
|---|---|---|
| FR-1.1 | 定义平台中立的声明式配置协议（YAML/JSON 表达），核心概念控制在少量、直觉化的名词内（初步设想：`Gateway`（实例/全局）、`Listener`（入口端口与 TLS）、`Route`（匹配与转发规则）、`Upstream`（后端服务与负载均衡）、`Policy`（限流、鉴权、CORS、重试等可插拔策略）） | P0 |
| FR-1.2 | 协议覆盖常见反向代理能力：域名/路径/header 匹配、路径重写、TLS 终止（含 SNI 多证书）、负载均衡策略、健康检查、超时/重试 | P0 |
| FR-1.3 | 协议覆盖常见 API 网关能力：限流、JWT/外部鉴权、CORS、请求/响应头改写 | P1 |
| FR-1.4 | 协议具备**扩展机制**：无法被抽象表达的能力可通过"escape hatch"（内嵌原生 Envoy 片段/patch）实现，不阻塞用户 | P1 |
| FR-1.5 | 协议版本化（如 `apiVersion`），保证向后兼容与演进能力 | P0 |
| FR-1.6 | L4（TCP/UDP proxy）转发支持 | P1 |
| FR-1.7 | 协议规范文档 + JSON Schema，可脱离本软件被第三方实现 | P1 |

### 5.2 FR-2 原生 Envoy 配置支持（专家模式）

| ID | 需求 | 优先级 |
|---|---|---|
| FR-2.1 | 用户可直接提交完整原生 Envoy static 配置，管理面负责语法/语义校验、存储、版本化与下发 | P0 |
| FR-2.2 | 抽象协议与原生配置可**混用**（如：整体用抽象协议，个别 listener 用原生片段） | P2 |
| FR-2.3 | 提供"查看编译产物"能力：抽象协议 → 生成的 Envoy 配置可随时导出查看，帮助用户学习和排障 | P0 |

### 5.3 FR-3 配置下发（驱动 Envoy）

| ID | 需求 | 优先级 |
|---|---|---|
| FR-3.1 | **static YAML 模式**：渲染完整 Envoy bootstrap/static 配置文件，协调 Envoy 热重启（hot restart）或依赖文件级动态资源（如 SDS/路由文件热加载）完成变更 | P0 |
| FR-3.2 | **xDS 模式**：内置 xDS 控制面（ADS，基于 go-control-plane 或等价实现），Envoy 以最小 bootstrap 接入，全部资源动态下发，变更零重启 | P0 |
| FR-3.3 | 配置变更原子性：一次变更要么整体生效要么整体不生效，下发前完成全量校验 | P0 |
| FR-3.4 | 配置版本管理：历史版本留存、diff 查看、一键回滚 | P1 |
| FR-3.5 | 管理 Envoy 进程生命周期（可选托管模式）：管理面可负责启动/守护本机 Envoy；也支持"仅下发"模式对接外部已运行的 Envoy | P1 |
| FR-3.6 | 多 Envoy 实例接入：一个管理面驱动多个对等 Envoy 节点（同一份配置，多点部署） | P2 |

### 5.4 FR-4 Web 管理控制台

| ID | 需求 | 优先级 |
|---|---|---|
| FR-4.1 | 配置管理 UI：以抽象协议模型为核心的表单化/可视化编辑，支持创建、修改、删除、发布 | P0 |
| FR-4.2 | 配置源码视图：YAML 编辑器（语法高亮、schema 校验提示），与表单视图双向切换 | P1 |
| FR-4.3 | **实时生效状态视图**：从 Envoy admin API 拉取并展示当前实际生效的 listeners、clusters、endpoints（含健康状态）、routes、certificates | P0 |
| FR-4.4 | 统计信息：请求量、成功率、延迟分布、连接数、按 route/cluster 维度的流量视图（基于 Envoy stats） | P1 |
| FR-4.5 | 配置发布流：变更预览（diff）→ 校验结果 → 发布 → 生效确认 | P1 |
| FR-4.6 | 控制台自身鉴权：本地账号 + 会话管理（首期），OIDC 对接（远期） | P0（基础）/ P2（OIDC） |
| FR-4.7 | 审计日志：谁在何时改了什么配置 | P2 |

### 5.5 FR-5 部署与运维

| ID | 需求 | 优先级 |
|---|---|---|
| FR-5.1 | Linux 原生部署：单二进制管理面 + systemd unit 文件，提供安装脚本/包（deb/rpm 远期） | P0 |
| FR-5.2 | Docker 部署：官方镜像，单条 `docker run` / 一份 `docker-compose.yml` 即可运行完整网关 | P0 |
| FR-5.3 | Kubernetes 部署：提供 Deployment/Helm chart 方式运行，**不依赖任何 CRD** | P1 |
| FR-5.4 | 管理面轻量化：低内存占用、无外部强依赖（默认内嵌存储如 SQLite/bbolt，配置以文件为真源可 git 管理） | P0 |
| FR-5.5 | 管理面自身可观测：健康检查端点、结构化日志、Prometheus metrics | P1 |
| FR-5.6 | 管理面故障不影响数据面：管理面宕机时 Envoy 按最后有效配置继续服务 | P0 |

### 5.6 FR-6 环境感知增强（k8s 可选能力）

| ID | 需求 | 优先级 |
|---|---|---|
| FR-6.1 | 自动检测是否运行于 k8s（ServiceAccount / 环境变量探测） | P1 |
| FR-6.2 | k8s 内运行时：列出集群 Namespace/Service/Endpoint，配置 upstream 时可搜索选择，自动填充地址 | P1 |
| FR-6.3 | k8s 内运行时：upstream 支持引用 Service 并跟踪其 Endpoints 变化（EDS 动态更新） | P2 |
| FR-6.4 | 非 k8s 环境：upstream 支持静态 IP 列表、域名（DNS 解析，含 SRV 远期）| P0 |

## 6. 非功能需求

| ID | 需求 |
|---|---|
| NFR-1 | **易上手**：从下载到完成第一个反向代理配置 ≤ 10 分钟；抽象协议核心概念一页文档可讲清 |
| NFR-2 | **轻量**：管理面常驻内存目标 < 150MB；单二进制无运行时依赖 |
| NFR-3 | **可靠**：配置校验拦截非法配置；数据面可用性不受管理面可用性影响（见 FR-5.6） |
| NFR-4 | **性能**：数据面性能即 Envoy 原生性能，管理面不在数据路径上 |
| NFR-5 | **安全**：xDS 通道支持 mTLS；控制台强制鉴权；证书私钥加密存储或对接外部 secret 来源 |
| NFR-6 | **兼容性**：明确声明支持的 Envoy 版本区间，随 Envoy 上游演进有升级策略 |
| NFR-7 | **可移植**：管理面支持 linux/amd64、linux/arm64、macOS（开发用途） |

## 7. 开源协议

采用 **AGPL-3.0**：

- 允许自由使用、修改、商用部署；
- 任何分发的修改版本，**以及以网络服务形式对外提供的修改版本**，必须以 AGPL-3.0 开源全部源码；
- 由此在事实上杜绝"闭源二次封装后售卖"的行为（开源协议在法律上不能禁止售卖行为本身，但 AGPL 使闭源转售不可行）。

## 8. 关键风险与开放问题

| # | 风险 / 开放问题 | 初步想法 |
|---|---|---|
| R1 | 抽象协议的表达力边界：抽象过度→能力受限，抽象不足→回到 Envoy 复杂度 | 以 80% 常见场景为设计目标，20% 长尾走 escape hatch（FR-1.4）；协议设计单独出一份 system_design 文档并广泛对比 nginx / Caddy / Traefik / Gateway API 的模型取舍 |
| R2 | static 模式下的变更生效机制：Envoy hot restart 协调复杂 | 优先推荐 xDS 模式；static 模式明确文档化其生效语义（可能需要 envoy 进程托管，FR-3.5） |
| R3 | 管理面与 Envoy 的部署拓扑（同机？同容器？分离？）影响进程托管与状态采集设计 | system_design 阶段定义 2~3 种标准拓扑并分别支持 |
| R4 | 技术选型：管理面语言（Go 生态有 go-control-plane，是自然选择）、前端框架、内嵌存储 | 在 system_design 中决策 |
| R5 | Envoy 版本升级兼容性（xDS API、配置字段演进） | 锁定 Envoy LTS-ish 版本区间，CI 做多版本配置编译验证 |
